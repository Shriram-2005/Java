# The Object Class: The "Big Three" Methods

You are standing exactly on the bridge between Phase 2 (Object-Oriented Architecture) and Phase 3 (The Java Collections Framework).

You cannot store, search, or sort massive amounts of custom objects in Java if you don't understand the "Big Three" methods of the `java.lang.Object` class. Because every class you ever write secretly `extends Object`, every class inherits these three methods. If you don't override them, Java uses the default versions, which will instantly break your data structures.

Here is the complete, production-ready package on `toString()`, `equals()`, and `hashCode()`, including the ultimate interview trap that breaks `HashMaps`.

## 1. `toString()` (The Nametag)

By default, if you try to print an object (`System.out.println(myEmployee);`), Java doesn't look at the data inside. It just prints the Class Name and the hexadecimal memory address in the Heap.
**Default Output:** `Employee@7a81197d`

This is completely useless for debugging. You must override `toString()` to return a human-readable string representing the object's state.

```java
public class Employee {
    int id;
    String name;

    public Employee(int id, String name) {
        this.id = id;
        this.name = name;
    }

    @Override
    public String toString() {
        // Now, printing the object yields: "Employee [ID=101, Name=Alice]"
        return "Employee [ID=" + this.id + ", Name=" + this.name + "]";
    }
}
```

## 2. `equals(Object obj)` (The Identity Check)

We discussed this in Phase 1 with Strings. By default, the `equals()` method inherited from `Object` acts exactly like the `==` operator. It only checks if two remote controls are pointing to the **exact same memory address**.

If you create two different `Employee` objects with the exact same ID and Name, the default `equals()` will say they are `false` because they live in two different boxes in the Heap.

To fix this, you override it to check the *actual data*.

```java
@Override
public boolean equals(Object obj) {
    // 1. Optimization: Are they literally the same memory box?
    if (this == obj) return true;
    
    // 2. Safety: Is the other object null, or a totally different Class?
    if (obj == null || this.getClass() != obj.getClass()) return false;
    
    // 3. Downcast: We know it's an Employee, so cast it
    Employee otherEmployee = (Employee) obj;
    
    // 4. The actual data check: Do they have the same ID?
    return this.id == otherEmployee.id; 
}
```

## 3. `hashCode()` (The Sorting Bucket)

This is the most misunderstood method in Java.
A Hash Code is simply an integer number generated by an algorithm.

Imagine you have 10,000 physical employee files, and you need to find "Alice" quickly. If they are in a massive pile, you have to look at every single one (O(n) time). But what if you have 26 buckets (A-Z)? You run Alice's name through a "Hash Function" (First letter = A). You go directly to Bucket A. You just saved massive amounts of time!

This is how Java's `HashMap` and `HashSet` work. `hashCode()` tells Java which "bucket" in memory to drop the object into.

* **Default Behavior:** Returns a random integer based on the physical Heap memory address.

```java
@Override
public int hashCode() {
    // We generate a hash code based strictly on the ID.
    // If two employees have the same ID, they get the exact same Hash Code!
    return Objects.hash(this.id); 
}
```

## 4. The Golden Contract (The Ultimate Interview Trap)

This is a guaranteed interview question: *"What is the contract between equals() and hashCode()?"*

> [!CAUTION]
> **THE RULE:** If two objects are equal according to `equals()`, they **MUST** return the exact same `hashCode()`.

**The Trap:** What happens if you override `equals()` to check the ID, but you forget to override `hashCode()`?

1. You create `Employee(101, "Alice")` and put it into a `HashSet` (A collection that prevents duplicates).
2. Because it uses the default memory-based `hashCode`, Java puts it in Bucket 5.
3. Later, you create a new `Employee(101, "Alice")` and try to add it. You expect the `HashSet` to reject it as a duplicate.
4. BUT, because it's a new object, the default `hashCode` drops it into Bucket 12.
5. The `HashSet` never even uses `equals()` to compare them, because they are in entirely different buckets! You now have a duplicate in your unique set, and your data is corrupted.

You must always override them together.
